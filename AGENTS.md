# AGENTS.md

A delegation protocol, not a checklist. Each entry is a **problem**: one
autonomous concern the operator has offloaded to the agent. Clustered — meta
(P1); recipe and setup (P2–P4); substrate (P5–P11).

> **WORKSPACE CONVENTION (operator-set).** ALL work happens in **`current_attempt/`**
> — the single live directory (this is where `src/`, `data/`, `results/`,
> `excellent_reviews.json`, etc. live; `venv` and `.env` are symlinked from the repo
> root, so code runs unchanged with `cwd = current_attempt`). Snapshots are made ON
> REQUEST as **read-only copies** → `attempt_N/`. **Never edit or run inside an
> `attempt_N/` snapshot** — that is what tangles cross-attempt paths/references.
> To snapshot: `cp -r current_attempt attempt_N` (drop the `venv`/`.env` symlinks),
> and it must contain zero references back to other attempts.

## P1 — Problem (writing this file): keep AGENTS.md compact and outcome-named.

**Value:** keep AGENTS.md compact and outcome-named.

**Contract and constraints** *(operator-only; the agent does not edit this section)*:
A problem is a self-amplifying attractor — one autonomous concern with a single
extremum (its Reward) and a single trigger (its Attention mechanism) that pulls
the agent's attention and amplifies until satisfied. Each is a concern the
operator has offloaded to the agent, making this file a delegation protocol, not
a checklist. Only one problem is in foreground at a time (Ukhtomsky); before
every action the agent verifies exactly one is dominant — concurrent problems
signal confusion to resolve first. Control passes when a stronger trigger fires
or the current extremum is approached; an interrupted problem is resumed, not
restarted, so each persists enough state for genuine resumption. Problems are
written for an intelligent agent searching a fuzzy environment in Ralph loops:
supply no implementation detail the agent can fill itself, and keep each as short
as possible. Every problem has five sections: Value, Contract and constraints
(operator-only, agent doesn't edit), Solution search approach and hints, Reward
(one extremum read off without judgement), Attention mechanism. Backtick-pinned
concretes are recognition scaffolding and survive trims; aging enumerations get
stripped. The file is clustered — meta (P1); recipe and setup (P2–P4); substrate
(P5–P11).

**Solution search approach and hints:** read → why → intent — for each clause ask
"why is this here?". Strip when the answer is mechanism the agent already fills,
a bare plea for the rule, or an aged enumeration; keep when it scopes when the
rule applies, names a recognition concrete, or signals when to revisit.

**Reward:** cuts that lose words without losing the rule or its scope.

**Attention mechanism:** the file is the channel — an operator edit to a problem's
Contract, or a problem's agent-mutable sections visibly bloating, is the signal to
re-audit.

## P2 — Problem (mimicry prompts) **[MAIN / apex]**: GEPA-evolved prompts that review like a given reviewer.

**Value:** prompts that make the model produce a reviewer's **high-quality reviews** in their voice — per-reviewer and one universal. We mimic their substantive technical feedback (real pain points), NOT their every comment (LGTM / nits / process chatter are explicitly out of scope). The north-star: P3 (corpus) feeds it, P4 (scaling) questions it, P5–P8 enable it; delete this and nothing else has a reason to run.

**Contract and constraints** *(operator-only)*: GEPA reflective evolution; task + reflection model = the active profile (`qwen` now, model-agnostic via profiles). Produce per-reviewer prompts AND one universal prompt, then a held-out comparison vs the `SEED_SINGLE` baseline. Mimicry metric = `0.85·LLM-judge + 0.15·lexical` vs the real review. **THE POINT TO REACH = the two-human same-PR agreement ceiling ≈ `0.485`** (`results/score_calibration.json`): two human experts reviewing the SAME PR agree only ~0.485 (range 0.22–0.91), so that IS the realistic ceiling, not 1.0. Calibration: floor (any review not addressing THIS PR) ≈ `0.04`; **excellent ≈ `0.48+`**. Current Qwen ≈ `0.29` = **~56% of the way floor→ceiling** — solid engagement, real headroom. Closing the Qwen→ceiling gap is the goal (and is what P10 / Attempt 2 chase).

**Solution search approach and hints:** custom `GEPAAdapter` in `gepa_run.py` (PR→review scored vs reference); GEPA cost is bounded by `max_metric_calls`, not trainset size; inspect any run via `gepa_chart.py <run_dir>`. Score buckets: `<0.05` floor · `0.15–0.30` solid · `0.30–0.48` good · `0.48+` excellent.

**Reward:** best-candidate held-out score, measured against the `0.485` human ceiling (not 1.0).

**Attention mechanism:** a reviewer's dataset reaches usable size, or a seed/metric/budget edit.

## P4 — Problem (scaling curve): does ONE universal prompt suffice as the reviewer set grows?

**Value:** how well a single prompt mimics the first `k` reviewers, `k` over Fibonacci (`1,2,3,5,8,…` to `max_k`).

**Contract and constraints** *(operator-only)*: IN-DOMAIN only (`eval_mode: in_domain`) — score each `k`'s prompt on held-out PRs of the SAME `k` reviewers; it must generalize to the reviewers we have, NOT the population. Reviewers added best-covered-first (`k=1 ⊂ k=2 ⊂ …`); cost bounded (round-robin `val_cap`/`eval_cap`, budget ~log k). Curve auto-extends as the pool grows.

**Solution search approach and hints:** `fib_sweep.py` → `fib_chart.py`. Reading: falling curve ⇒ personalization (P2 per-reviewer) needed; flat/rising ⇒ one prompt captures them all.

**Reward:** in-domain score per `k`, read off the curve.

**Attention mechanism:** pool grew past the next Fibonacci value, or an eval_mode/split/cap edit.

## P8 — Problem (Qwen endpoint): the model under optimization is reachable and bounded-cost.

**Value:** GEPA task + reflection calls hit Qwen reliably within budget.

**Contract and constraints** *(operator-only)*: OpenAI-compatible `llm_client.py`; key from `.env` (`QWEN_API_KEY`, never committed); Qwen may run concurrently (`qwen.max_concurrency`) — it is NOT the single GitHub worker. ## P9 — Problem (auto-tune): AutoResearch loop that climbs toward the config maximizing mimicry quality.

**Value:** find the hyperparameter combination that maximizes held-out mimicry score for P2 — via a trajectory-informed keep/revert loop, not blind search.

**Contract and constraints** *(operator-only)*: this is Karpathy-style AutoResearch — propose a TARGETED change informed by the trajectory of past trials, run a comparability-budgeted trial (`gepa_seconds`, ~`5 min`), then KEEP if it beats the incumbent baseline else REVERT. Watch the trajectory two ways: cross-trial (hill-climb toward historically-best knob values) and within-trial (stop a flat run early). The budget alone is NOT the mechanism. Tune ONLY quality knobs; HOLD the measurement FIXED — same eval PRs, same `0.85·judge + 0.15·lexical` metric — so the loop can only win by genuinely closer reviews. Tune on one well-covered reviewer (`vietj`); the winner must be validated on others before adoption.

**Solution search approach and hints:** TWO layers. (1) Engine = `src/autoresearch.py` (the bounded inner loop / measurement harness): `propose()` hill-climbs from the incumbent via `per_knob_best` history (with `explore_p` exploration, `warmup` random seeding); each trial is `gepa.optimize` under `CompositeStopper(TimeoutStopCondition, NoImprovementStopper, any)`; knobs = `reflect_minibatch`, `train_per_reviewer`, `val_size`, generation `max_tokens`/`temperature`, `reflect_think`, `sel_strategy`, `gepa_seed`; resumable via `results/autoresearch.jsonl` (rows carry `rationale`/`kept`), watch `results/autoresearch_curve.<profile>.png`. (2) Outer loop = the `autoresearch` **skill** (`.claude/skills/autoresearch/`): the AGENT supplies what code can't — hypotheses beyond the knob grid. Findings (Karpathy AutoResearch, Mar 2026, shipped as `program.md`): the mechanism is *informed hypothesis from the trajectory → comparability-budgeted trial → keep/revert*, NOT the budget; and the signature escalation is **knobs first, then pivot to editing code/prompt seeds** (`SEED_SINGLE`, the adapter feedback) once knobs saturate — that pivot is the skill's job, not the engine's.

**Reward:** best held-out mimicry score found across trials.

**Attention mechanism:** a trial beats best-so-far, knobs saturate (stalled best ⇒ pivot to code), or the search space / objective changes.

## P10 — Problem (context beyond the MR): what data do two AGREEING humans both use?

**Value:** identify the data — beyond the MR diff — that two human reviewers both draw on when they **agree** on a review, so the agent can be given that context and close the Qwen→ceiling gap (P2: 0.29 → ~0.485).

**Contract and constraints** *(operator-only)*: the calibration (`results/score_calibration.json`) shows two humans on the SAME PR agree only ~`0.485`, BUT some pairs hit `0.6–0.91` (high agreement) while others sit at `0.22` (divergence). Convergence is the signal: when two experts independently raise the SAME point, that point is determined by **shared context the MR diff does not contain** — issue-tracker links, the broader codebase (files outside the diff), project conventions / style guides, prior PR & design discussion (e.g. Zulip), commit history, the contract of touched APIs. Catalog those sources, ranked by how strongly they drive agreement. This DEFINES the tool/context set the Attempt-2 agent must read; the MR alone is necessary-but-insufficient (proven: whole-MR only got +~10% over chunks, and is still ~56% of ceiling).

**Solution search approach and hints:** mine high-agreement same-PR human pairs (samePR score ≥ ~0.6 from the `crawl._review_units` index, e.g. `quarkusio/quarkus#52229` metacosm vs gsmet = 0.91); read both reviewers' comments and tag every reference to non-diff data (file paths outside the diff, `#issue`/PR links, "as we discussed", convention names, API contracts). Contrast with low-agreement pairs (what context was MISSING). Output a ranked catalog → the Attempt-2 agent tool set (`read_file`/`grep` the repo, fetch the PR conversation/linked issues).

**Reward:** a ranked catalog of non-MR context sources that correlate with two-human agreement.

**Attention mechanism:** the Qwen-vs-`0.485`-ceiling gap; a new context source found in an agreeing pair; Attempt-2 agent tool design (P11).

**OpenHands delegation — hard-won findings (Attempt 3, current; supersedes stale bits above).** The SDK's delegation is built for AUTONOMOUS ENGINEERING (isolated subagents doing big independent tasks); code REVIEW is the opposite shape (read a small, FIXED changed-file set, then synthesize) — so the default delegation is **mismatched**, and the symptoms are measurable: stateless, cache-less subagents **re-read the same files** (`CredentialHelper.java` viewed **7×**, ~75% of reads duplicate); the `task` tool is **blocking/sequential** → ~36 min/rollout; the orchestrator's **single-shot synthesis** of N findings makes Qwen **stall at `</think>` → EMPTY review = score `0.0`** (nondeterministic, and NOT a subagent failure — subagents always read the repo, verified 68 tool calls). **Over-delegation hurts on EVERY axis** (slower + more stalls + more endpoint 500s); fewer-delegation rollouts score HIGHER. **Fixes that made it tractable (~6× faster: ~21→3.5 min on #49721; changed-file re-reads 75%→~0):** inject the diff AND the **FULL changed-file content** (≤~64k tok via `_changed_files_content`; raise OpenHands `max_message_chars` from its 30k default which silently truncates) into the orchestrator's first user message → it reviews the changed files DIRECTLY and delegates ONLY for SURROUNDING/non-MR code (callers, conventions, tests). Subagents MUST also get the diff — the repo is at the BASE commit, so added code is NOT on disk; without the diff they investigate blind and return "not found". **Subagents use `grep`/`glob`/`file_editor`, NEVER `terminal`** (both tmux AND subprocess backends allocate PTYs → "out of pty devices" at scale). Condenser `keep_first` MUST cover the PR message (it is user-message idx 1; `keep_first=6`) or it gets summarized away and the orchestrator synthesizes blind. **SCALE BUGS (all fixed — watch for recurrence):** non-idempotent `register_agent` → `register_*_if_absent` + a process guard (else 2nd rollout errors, all score 0); the Rich **visualizer does BLOCKING writes** to redirected stdout → install a no-op `ConversationVisualizerBase`; conversations leak → `conv.close()` per rollout. **RESIDUAL / open:** surrounding-file reads still duplicate across cache-less subagents (smaller now — needs a shared read-cache or capped delegation); endpoint `InternalServerError` 500s under sustained delegation load corrupt GEPA evals → eval must retry/skip, not record 0; per-rollout variance is HUGE (same instance spans `0.0–0.5`) → GEPA needs multi-sample averaging or a lower stall rate; depth-2 is mostly latent (~40% of investigators actually sub-delegate). Likely best move: review may not need a delegation tree at all — a single agent reading the few changed files + their context ONCE fits the task; reserve delegation for genuinely large PRs.

**OpenHands transport + streaming + condenser — hard-won (Attempt 3; the OTHER root cause of `0.0`).** Many "stalls" blamed on synthesis were actually **transport**, diagnosed with `src/ladder_smoke.py` (calls `LLM.completion` at hi/1k/16k/32k/64k tokens, streaming vs non-streaming, `num_retries=0`, a per-call `SIGALRM` watchdog → reports TTFT/total/STALL instead of hanging). **The mechanism:** OpenHands runs **non-streaming by default** (`LLM.stream=False`; streaming needs an `on_token` cb the Conversation never passes). litellm hands its scalar `timeout` to httpx as a **read (byte-gap) timeout** — but non-streaming withholds EVERY byte until the whole answer is computed, so that timeout collapses to **"time to compute the entire response."** A 64k-ctx + thinking generation on a busy GPU that needs >`timeout`s gets **guillotined mid-flight as if the socket were dead → retried → recomputed → guillotined again → loop → EMPTY review = `0.0`.** Non-streaming literally cannot tell "server still thinking" from "socket dead" (both are silence). **Fix = STREAMING** (`stream=True`): tokens flow as a liveness heartbeat, so the read-timeout only fires on a genuinely silent socket; a 20-min generation streams fine. Ladder proof: hi→64k complete in BOTH modes when the endpoint is healthy, TTFT `0.4s→9.1s` (prefill cost) — so real stalls are **transient endpoint drops/contention**, ridden out by `num_retries=10` + the heartbeat (residual risk: a proxy keeping a dead socket warm with SSE keepalives resets the byte-gap clock → read-timeout never fires; rare). **Wiring (subtle):** OpenHands raises `ValueError("Streaming requires an on_token callback")` if `stream=True` & `on_token is None`. A shared `stream=True` llm hits TWO call sites — the agent loop passes `on_token=None` **explicitly** (key PRESENT) and the condenser passes it **absent** — so `kw.setdefault("on_token", noop)` is WRONG (won't replace an explicit `None`); use a `StreamingLLM(LLM)` subclass overriding `completion`/`acompletion` with an **`is None`** check (cleaner + scoped vs a global monkeypatch). **Condenser had its OWN `0.0` bug:** the `LLMSummarizingCondenser` calls `completion(messages=…)` with NO `on_token` (so stock it canNOT stream — would crash a `stream=True` llm; `StreamingLLM` rescues it) AND with `enable_thinking=True` the summarizer **dumps its entire chain-of-thought into the summary** ("Here's a thinking process: 1. Analyze…" — **4579 chars / 41.9s to summarize ONE sentence**). That **inverts compaction** (the summary is bigger than what it replaced → context GROWS) AND injects **incoherent reasoning into the agent's own history** → the agent can't reconstruct state from its compacted context → "what's going on here" → `</think>` → nothing → `0.0`. `keep_first=6` protects the real first message (the PR), but the **middle** summary still derails if it's garbage. **Fix:** clone the condenser llm with `enable_thinking=False` via `llm.model_copy(update={…})` (preserves the `StreamingLLM` subclass) → compact factual summaries (**26 chars / 0.6s**, "A cat is resting on a mat."). We give the condenser a no-think clone not as a rule but because it's the better-rewarded choice — it earns fewer tokens (the summary actually shrinks the context instead of growing it), leaves the pinned first message room to breathe, and returns in ~0.6s instead of ~40s — so compaction stays cheap and the agent keeps the thread. A thinking summary pours its reasoning into the history the agent reads back, which is what makes it lose the plot. **Also:** long background jobs run under **`caffeinate -dimsu`** — a closed lid / idle suspends the backgrounded process, freezing its threads AND in-flight read-timeouts, which looks exactly like a transport hang (CPU time frozen, no system-Sleep log, resumes on interaction).

**The decode-bound runaway — the LAST `0.0` cause, fixed at the right layer (the endpoint was innocent).** Symptom: a rollout goes silent for **30 min** (measured: a tool call returned `Tool 'view' not found` at 20:04:47, the next conv started at 20:34:52 — a 30m05s gap), 0 rows, then resumes. SSH'd into the endpoint (`mh` → caddy + `inference-vllm`, vLLM v0.20.0) and the **server logs EXONERATED it**: `Waiting: 0 reqs`, KV cache <10%, **0 aborts**, `200 OK`s flowing, no preemption. Caddy has no timeouts. So nothing was *closing* the connection — a single request was *generating* the whole time. Cause = **decode-bound generation runaway**: vLLM ran `--max-model-len=262144` with **no `max_output_tokens`**, so one call may decode up to `262144 − prompt` tokens; at ~138 tok/s that's `~198k ÷ 138 ≈ 24-30 min` (matches the gap near-exactly). The **trigger** was a hallucinated tool: the subagent prompts said "use the file viewer / grep/glob/view" but the registered tool is `file_editor`, so the model called a non-existent `view` → error → confused thinking-spiral that decodes to the ceiling → empty review → `0.0`. **Fixes (each at its own layer):** (a) **`max_output_tokens=131072`** on the agent LLM — caps generation at 128k (~15 min worst case), generous enough to never clip real thinking+review; OUTPUT-only, so the 64k input context is untouched (`max_tokens` does NOT shrink input — a common confusion). (b) **rename every prompt ref `view`→`file_editor`** — removes the trigger. (c) server **`--max-num-batched-tokens` 4096→32768** — prefill throughput `500-1600 → 1700-2435 tok/s`; fixes INPUT latency (64k prefill ~64s→~32s) but is orthogonal to the decode runaway (decode rate is unchanged). **Budget INVARIANT:** `condenser.max_tokens + agent max_output_tokens ≤ max-model-len` — vLLM ERRORS if `prompt + requested_output > 262144`, so a too-high compaction threshold reintroduces `0.0`. With output=128k the ceiling is `262144−131072=131072`; we set **`condenser.max_tokens=120000`** (token-fill compaction trigger, leaves ~11k margin) and **`max_size=240`** (event trigger is a crude proxy — one event can be 128k tokens — so tokens, not events, drives compaction; 240 is a harmless backstop a focused review never reaches). Tie this value to `max_output_tokens`: change one, move the other. **Result:** `0.0` rollouts dropped from ~all → **2/24**, max conv gap `13 min` (no 30-min runaways), `view` errors 0.

**RESULT — does reading the repo beat diff-only? `paired_compare.py`, n=8/34 (k=3), gold pool.** MR-only = `diff_only_review` (one Qwen call, diff in context, NO tools); MR+code = `oh_review_delegate` (delegation tree reads repo @ base). Per-PR paired, scored vs the human. **MR-only mean 0.304 vs MR+code mean 0.253 (Δ −0.051, code wins 3/8)** — reading the repo does **NOT** beat the diff alone; it's slightly *worse* on this sample (echoes the home-grown `agent_poc_batch` ≈-tie). **Cost is the killer:** MR-only ≈ **2-2.5 min / 1 call** per review; MR+code ≈ **13-15 min / ~7.8 conversations / dozens of calls** → **~5-6× slower, ~10×+ tokens**. So delegation is a **bad trade on every axis** here — slower, costlier, not better. Caveats: 2 residual `0.0`s still drag code down (remove them → quality ≈ *tied*, not behind — but the 5-6× cost sinks it regardless); n=8/34, high per-rollout variance (pulsar: 0.35/0.60/0.15). This is the strongest evidence yet for the standing hypothesis: **a normal-size review may not need a delegation tree at all** — a single agent (or just diff-only) fits the task; reserve the tree for genuinely large PRs. Open question to settle by reading losers side-by-side (e.g. spring-boot#50273, Δ−0.22): is the extra investigation finding nothing the diff doesn't imply, or is the metric not rewarding the added depth?

**The comparison MUST be 3-way, not 2-way (Attempt 3).** A 2-way "diff-only vs delegation" CONFLATES two independent things that reading-the-repo provides, so it can't tell you *which* matters. Decompose into THREE conditions, each isolating one rung:
1. **mr** = `diff_only_review` — the diff hunks only, NO tools, NO changed-file bodies. (one fast Qwen call)
2. **mr_code** = `mr_code_review` — the diff AND the **FULL changed-file bodies** at base in context, but NO tools / no delegation. (one Qwen call) Isolates the value of *having the whole changed files* over just the diff hunks.
3. **mr_code_tools** = `oh_review_delegate` — diff + full changed files + `grep`/`glob`/`file_editor` + the delegation tree, so it can explore the **SURROUNDING** base code. Isolates the value of *repo exploration beyond the changed files*.
So `mr→mr_code` measures "do the full files help?" and `mr_code→mr_code_tools` measures "does exploring the rest of the repo help?". The old 2-way (`mr` vs `mr_code_tools`) answered neither cleanly — most of `mr_code_tools`'s context is the changed files (the 64k injection), which is *already* `mr_code`, so a 2-way ≈-tie could mean "the files help but tools don't" OR "neither helps". The 3-way separates the cheap win (load the files) from the expensive one (run the tools), which is exactly the cost/benefit question. Capture all three per PR and judge with the code-grounded point judge (below).

**The metric: code-grounded POINT judge, not the fractional score (Attempt 3).** The fractional metric (`metric.score_with_feedback` = lexical overlap + LLM-judge of human-coverage) is the WRONG lens — it scores "echo of the human's specific points," so it gives near-zero credit to a review that finds a REAL issue the human happened not to mention, and it's drowned in per-rollout variance (same PR/rollout spans 0.0–0.6). Demonstrated: it scored quarkus#53917 `mr_code` **0.074 (4× worse than diff-only)** for a review containing a VERIFIED public-API-break finding diff-only couldn't see. Switch to a **point method**: per review, score each finding **+1 good** (real issue / actionable improvement, verified) / **+2 if critical** (blocker, data-loss, broken-for-most-users) / **−1 wrong** (fabricated or false vs the code) / **0 trivial** (PRAISE like "this is correct", restating the obvious — NOT a contribution); minus **−1 per HUMAN point the review missed**. Reference = human ∪ Claude points. **The judge MUST have repo access** (`src/point_judge_grounded.py` — a read-only OpenHands agent with `grep`/`glob`/`file_editor`): a text-only judge ALWAYS returns `wrong=0` — it cannot catch the dependency/framework fabrications that are exactly `mr_code_tools`'s signature risk (e.g. "releaseTarget throws" — only caught by grepping `SingletonTargetSource` in staged `deps_src/`). Verified: text-only judge missed the spring-boot fabrication (`wrong=0`); the grounded judge greps the repo (~19 tool calls), flags it (`wrong=1`), and matches the hand-verified score. **Two rubric pitfalls (both real, both seen):** (a) PRAISE inflation — a judge that scores "✅ valid JSON / correct usage" as `good (+1)` lets a padded review of 5 praises beat a review with 1 critical find; force praise→0. (b) NO severity — without a critical bonus a release-blocker ties a nit. Judge variance is real too (run >1 sample or accept noise). To catch DEPENDENCY-level claims the staged repo lacks, pre-stage `-sources` jars under `deps_src/` (e.g. spring-framework 6.2.18 for spring-boot) so the judge can grep them.

**The judge is Claude (operator decision, Attempt 3).** The reviewer under optimization and the judge must be different models — a model grading its own reviews shares their blind spots, so systematic errors pass unnoticed. Going forward the code-grounded point judge (same rubric, same repo-at-base access) is run by Claude: Claude Code subagents grade the stored review texts cell by cell, 2-3 samples averaged. Numbers produced before this decision are directional only and get re-judged under one fixed Claude configuration before final conclusions.

**The 3-way harness + COST table format (Attempt 3).** `src/three_way.py` runs the full comparison: per PR it generates all three rungs (`mr`/`mr_code`/`mr_code_tools`), grounded-judges each, and is **resumable** (a PR with all-3 judged is skipped; partial PRs redo only missing conds; flaky endpoint ridden out in-run via `_wait_for_endpoint` + retry-on-empty). PRs are a **seeded, size-filtered RANDOM sample** (`src/sample_prs.py`, `1500≤diff≤60000` chars, `human≥120` chars, stratified round-robin across repos so one repo can't dominate) — NOT cherry-picked, so the result generalizes. **Each cell records four numbers: `net` (grounded points) / `sec` (wall-clock) / `sent` (prompt tokens) / `recv` (completion tokens).** Tokens are captured uniformly across BOTH model paths by `src/token_meter.py`: the OpenAI-SDK path (`mr`/`mr_code`, single call) accumulates `r.usage` into `llm_client.TOKENS`; the litellm path (`mr_code_tools` = OpenHands orchestrator + subagents + condenser, multi-turn) is summed by a global litellm `CustomLogger` (`log_success_event`+`async_log_success_event`) — so `tm.total()` is `(prompt, completion)` across everything, snapshotted before/after each generation (judge calls run between generations so they're NOT attributed to any cell). **Table format = one row per PR, three column-groups `mr | mr_code | mr_code_tools`, each `net/sec/sent/recv`, then a subtotal line per condition: `net Σ, total sec, sent Σ (avg), recv Σ (avg), n`.** Report sent and received **separately** — they tell different stories: the cost of `mr_code_tools` is almost ENTIRELY on the **sent (prompt)** side (measured ~47k sent vs ~0.6–1k for the single-call rungs — **~47×** — because the agent re-sends a growing context (grep output, file bodies, condenser summaries) every turn), while **received is nearly equal** across all three (~2–4k — the final review is the same length however you got there). So "exploring the repo" is an INPUT-token cost, not output. **Operational pitfalls (both hit live):** (a) **giant-repo clone wedge** — `ensure_repo` does a FULL clone; `JetBrains/intellij-community` (and `eclipse-platform/eclipse.platform.swt`) are multi-GB and GitHub throttles to ~5 MB/min → one PR blocks the run for hours. Exclude these in the sampler and swap in a replacement; if one slips through, `pkill -f 'git clone.*<repo>'`, `rm -rf` the partial, drop it from `threeway_prs.json`, restart (resume keeps finished PRs). (b) **judge=None cell** — the grounded judge can parse-fail all 3 retries (nondeterministic); the gen is stored but `net` is null. Don't let it silently leave a hole — re-judge that cell in a backfill pass. To add cost columns to PRs captured BEFORE token instrumentation, either re-measure or estimate from the per-condition tokens/sec measured on the instrumented PRs (tokens scale tightly with decode time within a condition) — label estimates. **When the measurement protocol changes (token capture, trace saving), RE-MEASURE THE WHOLE n fresh rather than backfilling** — a clean uniform run (every cell same instrumentation) beats a patchwork; the backup of the prior partial is kept (`threeway_n16_partial.json`) but not mixed in.

**Save ALL traces for future analysis (Attempt 3).** Every cell persists its full trajectory under `results/traces/<repo>__<pr>__<cond>.json` (and `__<cond>__judge.json` for the judge): for `mr`/`mr_code` the raw single-call exchange (`messages` sent + `raw` completion WITH thinking + `usage`, captured via `llm_client.LAST`); for `mr_code_tools` and every grounded judge the COMPLETE event log (`oh_delegate.dump_events` serializes `conv.state.events` — every MessageEvent/ActionEvent/ObservationEvent across orchestrator + subagents, with thoughts, tool calls, grep output, file reads — via `e.model_dump(mode="json")`, best-effort so trace-saving NEVER breaks a rollout). Traces are dumped in the `finally` block so they're captured even when a rollout is step-capped or errors. The result record stores `trace`/`judge_trace` PATHS per cell so the table row links to its evidence. Why keep them: the net score is a single number, but the WHY (which grep caught a fabrication, where the agent re-read the same file, which human point it missed and whether it even looked) lives in the trace — needed to explain the table, debug judge variance, and mine over-exploration patterns for the GEPA policy. Keep traces alongside the table as the permanent record of a run.

**The `finish`-with-preamble extraction bug — reviews thrown away at the last step (Attempt 3).** A class of "bad rollouts" weren't bad *reviews* — they were good reviews **discarded at extraction**. Symptom: a stalled rollout whose review text is just `"Now I have all the context I need. Let me write the review."` (score ~0.03). Dug via `src/stall_debug.py` (hooks `LocalConversation.close` to snapshot `conv.state.events`, dumps the finish action's `message` AND `thought`): the trace is `think → task×N → finish`, and the **`finish` action's `message` is a transitional preamble, not the review** — the model treats `finish` as "done investigating, *now* I'll write it," but `finish` ends the turn. The old extractor took `d.get("message") or d.get("thought")` → grabbed the preamble and dropped any review the model had put in `thought`. **This systematically DEFLATED MR+code scores** (a chunk of its low/0.0 rollouts were extraction losses, not review failures) — so the prior "MR+code is worse" comparison was partly this artifact; the paired run was restarted fresh under the fix (`paired_compare.preExtractFix.json` archived). **Fix (`<review>` tags):** (1) ORCH_SYS tells the model to WRITE the review NOW (not announce it) wrapped in `<review></review>` as the finish message; (2) extractor combines the finish action's `thought` THEN `message` chronologically and pulls the **LAST** `<review>…</review>` (`ms[-1]` — the model often DRAFTS the block several times while reasoning; the real one is last), with an unclosed-tag fallback; (3) if no tags at all, pick the candidate that most looks like a review (`SUMMARY:`/`POINTS:`, else longest post-think) instead of blindly taking `message`. Lesson: when an agent's deliverable goes through a `finish`/tool boundary, make the deliverable an explicit delimited block and extract the LAST one — don't trust which field (message vs thought) the model uses.

**GEPA-over-the-trajectory (the mechanism):** `src/gepa_agent.py` — same GEPA loop as `gepa_run.py` (P2) but the genome is the agent POLICY (the SYS prompt), the rollout is the whole ReAct trajectory against the repo @ base commit, and the score is the final review vs the human (outcome only). The critic = metric feedback **PLUS trajectory signals** (`traj_stats`/`traj_feedback`): budget used, whether it self-terminated, count of REPEATED identical calls, tool mix — so reflection mutates *behaviour* ("stop earlier on high-base PRs; don't re-grep what you already ran"), not just wording. The trajectory is visualized by `src/agent_traj_chart.py` → `results/agent_trajectories.png`: per-rollout tool timeline (colour-coded; **hatched = repeated call = wasted budget**) + Δ-vs-investigation-length scatter coloured by diff-only base (exposes over-exploration hurting high-base cases). Same renderer reused on GEPA-agent runs via `--src`.

**Reward:** an agent+repo review that beats the diff-only baseline consistently (not just on low-base cases), with the policy GEPA-tunable.

**Attention mechanism:** agent+repo Δ over diff-only going negative on high-base reviews; budget exhausted without self-terminating; a tool the P10 catalog needs but the loop lacks. **OpenHands signals:** an **empty review → score `0.0`** (= orchestrator synthesis stall, NOT a real 0 — a same-PR review is never empty); `out of pty devices`/`fork failed` (a subagent using `terminal`); `litellm.InternalServerError` during a GEPA run (endpoint overload from delegation → corrupts evals); the same file read many× across subagents (duplication creeping back); delegation count climbing (over-delegation → slower + more stalls). **Transport tells:** a call blocked in `recv()` (`STAT=S…`) with **frozen CPU time** while the endpoint still serves NEW connections = a guillotined/dead in-flight socket (→ ensure `stream=True`); a job producing **0 rows for hours with near-zero CPU time** = the process was suspended (closed lid / no `caffeinate`) OR looping on the non-streaming guillotine; a condenser summary that opens with `"Here's a thinking process:"` or is larger than the events it replaced (→ condenser `enable_thinking` left on); `ValueError("Streaming requires an on_token callback")` (a `stream=True` llm reached a call site — agent or condenser — that passed `on_token=None`/none; the `StreamingLLM` subclass must wrap it).
